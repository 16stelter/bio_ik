3 ## Disclaimer
4 
5 This repository provides a BSD-licensed standalone implementation
6 of a variety of optimization methods to efficiently solve generalized inverse kinematics problems.
7 
8 The whole module was implemented by Philipp Ruppel as part of his Master Thesis.
9 
10 For a C++-based reimplementation of the original "BioIK" algorithm,
11 as originally sold in the Unity store, you can use the non-default mode `bio1`.
12 The default mode `bio2_memetic` shares no code with this implementation, was shown to outperform it
13 in terms of success rate, precision and efficiency, and is actually usable for precise robotic applications [4].

15 ## Installation and Setup
16 
17 You will need ROS version Indigo or newer (wiki.ros.org).
18 The software was developed on Ubuntu Linux 16.04 LTS with ROS Kinetic,
19 but has also been tested on Ubuntu Linux 14.04 LTS with ROS Indigo.
20 Newer versions of ROS should work, but may need some adaptation.
21 See below for version specific instructions.

29  roscd

38  the sources including a few private header files are in the `src` subdirectory.

41 ## Basic Usage
42 
43 For ease of use and compatibility with existing code,
44 the bio-ik algorithm is encapsulated as a Moveit kinematics plugin.
45 Therefore, bio-ik can be used as a direct replacement of
46 the default Orocos/KDL-based IK solver.
47 Given the name of an end-effector and a 6-DOF target pose,
48 bio-ik will search a valid robot joint configuration that reaches the given target.
49 
50 In our tests (see below), both in terms of success rate and solution time,
51 bio-ik regularly outperformed the Orocos [1] solver
52 and is competitive with trac-ik [2].
53 The bio-ik algorithm can also be used for high-DOF system like robot snakes,
54 and it will automatically converge to the best approximate solutions
55 for low-DOF arms where some target poses are not reachable exactly.
56 
57 While you can write the Moveit configuration files by hand,
58 the easiest way is to run the Moveit setup assistant for your robot,
59 and then to select bio-ik as the IK solver when configuring the end effectors.
60 Once configured, the solver can be called using the standard Moveit API
61 or used interactively from rviz using the MotionPlanning GUI plugin.
62 
63 * Make sure that you have a URDF (or xacro) model for your robot.
64 * Run the moveit setup assistant to create the Moveit configuration files:
65 
66  ```
67  rosrun moveit_setup_assistant moveit_setup_assistant
68 
69  ```
70 * The setup assistant automatically searches for all available IK solver plugins
71  in your workspace.
72  Therefore, you can just select select bio-ik as the IK solver
73  from the drop-down list for every end effector and then configure
74  the kinematics parameters, namely the default position accuracy (meters)
75  and the timeout (in seconds). For typical 6-DOF or 7-DOF arms,
76  an accuracy of 0.001 m (or smaller) and a timeout of 1 msec should be ok.
77  More complex robots might need a longer timeout.
78 * Generate the moveit configuration files from the setup assistant.
79  Of course, you can also edit the `config/kinematics.yaml` configuration
80  file with your favorite text editor.
81  For example, a configuration for the PR2 robot might look like this:
82 
83  ```
84  # example kinematics.yaml for the PR2 robot
85  right_arm:
86  # kinematics_solver: kdl_kinematics_plugin/KDLKinematicsPlugin
87  # kinematics_solver_attempts: 1
88  kinematics_solver: bio_ik/BioIKKinematicsPlugin
89  kinematics_solver_search_resolution: 0.005
90  kinematics_solver_timeout: 0.005
91  kinematics_solver_attempts: 1
92  left_arm:
93  kinematics_solver: bio_ik/BioIKKinematicsPlugin
94  kinematics_solver_search_resolution: 0.005
95  kinematics_solver_timeout: 0.005
96  kinematics_solver_attempts: 1
97  all:
98  kinematics_solver: bio_ik/BioIKKinematicsPlugin
99  kinematics_solver_search_resolution: 0.005
100  kinematics_solver_timeout: 0.02
101  kinematics_solver_attempts: 1
102 
103  # optional bio-ik configuration parameters
104  # center_joints_weight: 1
105  # minimal_displacement_weight: 1
106  # avoid_joint_limits_weight: 1
107  ```
108 
109 
110 * For a first test, run the Moveit-created demo launch. Once rviz is running,
111  enable the motion planning plugin, then select one of the end effectors
112  of you robot. Rviz should show an 6-D (position and orientation)
113  interactive marker for the selected end-effector(s).
114  Move the interactive marker and watch bio-ik calculating poses for your robot.
115 
116  If you also installed the bio-ik demo (see below), you should be able
117  to run one of the predefined demos:
118  ```
119  roslaunch pr2_bioik_moveit demo.launch
120  roslaunch pr2_bioik_moveit valve.launch
121  roslaunch pr2_bioik_moveit dance.launch
122  ```
123 
124 * You are now ready to use bio-ik from your C/C++ and Python programs,
125  using the standard Moveit API.
126  To explicitly request an IK solution in C++:
127  ```
128  robot_model_loader::RobotModelLoader robot_model_loader(robot);
129 
130  auto robot_model = robot_model_loader.getModel();
131  auto joint_model_group = robot_model->getJointModelGroup(group);
132  auto tip_names = joint_model_group->getSolverInstance()->getTipFrames();
133 
134  kinematics::KinematicsQueryOptions opts;
135  opts.return_approximate_solution = true; // optional
136 
137  robot_state::RobotState robot_state_ik(robot_model);
138 
139  // traditional "basic" bio-ik usage. The end-effector goal poses
140  // and end-effector link names are passed into the setFromIK()
141  // call. The KinematicsQueryOptions are empty.
142  //
143  bool ok = robot_state_ik.setFromIK(
144  joint_model_group, // joints to be used for IK
145  tip_transforms, // multiple end-effector goal poses
146  tip_names, // names of the end-effector links
147  attempts, timeout, // solver attempts and timeout
148  moveit::core::GroupStateValidityCallbackFn(),
149  opts // mostly empty
150  );
151  ```

153 ## Advanced Usage
154 
155 For many robot applications, it is essential to specify more than just
156 a single end-effector pose. Typical examples include
157 
158 * redundancy resolution (e.g. 7-DOF arm)
159 * two-arm manipulation tasks on two-arm robots (e.g. Baxter)
160 * multi end-effector tasks with shared kinematic links
161 * grasping and manipulation tasks with multi-finger hands
162 * full-body motion on humanoid robots
163 * reaching tasks with additional constraints (e.g. shoulder position)
164 * incremental tool motions without robot arm configuration changes
165 * and many more
166 
167 In bio-ik, such tasks are specified as a combination of multiple
168 individual *goals*.
169 The algorithm then tries to find a robot configuration
170 that fulfills all given goals simultaneously by minimizing
171 a quadratic error function built from the weighted individual goals.
172 While the current Moveit API does not support multiple-goals tasks directly,
173 it provides the KinematicQueryOptions class.
174 Therefore, bio-ik simply provides a set of predefined motion goals,
175 and a combination of the user-specified goals is passed via Moveit to the IK solver.
176 No API changes are required in Moveit, but using the IK solver now consists
177 passing the weighted goals via the KinematicQueryOptions.
178 The predefined goals include:
179 
180 * *PoseGoal*: a full 6-DOF robot pose
181 * *PositionGoal*: a 3-DOF (x,y,z) position
182 * *OrientationGoal*: a 3-DOF orientation, encoded as a quaternion (qx,qy,qz,qw)
183 * *LookAtGoal*: a 3-DOF (x,y,z) position intended as a looking direction
184  for a camera or robot head
185 * *JointGoal*: a set of joint angles, e.g. to specify a
186 * *FunctionGoal*: an arbitrary function of the robot joint values,
187  e.g. to model underactuated joints or mimic joints
188 * and several more
189 
190 
191 
192 To solve a motion problem on your robot, the trick now is to construct
193 a suitable combination of individual goals.
194 
195 ![PR2 turning a valve](doc/pr2_vt_0.png)
196 
197 In the following example, we want to grasp and then _slowly turn
198 a valve wheel_ with the left and right gripers of the PR2 robot:
199 
200  ```
201  bio_ik::BioIKKinematicsQueryOptions ik_options;
202  ik_options.replace = true;
203  ik_options.return_approximate_solution = true;
204 
205  auto* ll_goal = new bio_ik::PoseGoal();
206  auto* lr_goal = new bio_ik::PoseGoal();
207  auto* rl_goal = new bio_ik::PoseGoal();
208  auto* rr_goal = new bio_ik::PoseGoal();
209  ll_goal->setLinkName("l_gripper_l_finger_tip_link");
210  lr_goal->setLinkName("l_gripper_r_finger_tip_link");
211  rl_goal->setLinkName("r_gripper_l_finger_tip_link");
212  rr_goal->setLinkName("r_gripper_r_finger_tip_link");
213  ik_options.goals.emplace_back(ll_goal);
214  ik_options.goals.emplace_back(lr_goal);
215  ik_options.goals.emplace_back(rl_goal);
216  ik_options.goals.emplace_back(rr_goal);
217  ```
218 
219 We also set a couple of secondary goals.
220 First, we want that the head of the PR2 looks at the center of the valve.
221 Second, we want to avoid joint-limits on all joints, if possible.
222 Third, we want that IK solutions are as close as possible to the previous
223 joint configuration, meaning small and efficient motions. This is handled
224 by adding the MinimalDisplacementGoal.
225 Fourth, we want to avoid torso lift motions, which are very slow on the PR2.
226 All of this is specified easily:
227 
228  ```
229  auto* lookat_goal = new bio_ik::LookAtGoal();
230  lookat_goal->setLinkName("sensor_mount_link");
231  ik_options.goals.emplace_back(lookat_goal);
232 
233  auto* avoid_joint_limits_goal = new bio_ik::AvoidJointLimitsGoal();
234  ik_options.goals.emplace_back(avoid_joint_limits_goal);
235 
236  auto* minimal_displacement_goal = new bio_ik::MinimalDisplacementGoal();
237  ik_options.goals.emplace_back(minimal_displacement_goal);
238 
239  auto* torso_goal = new bio_ik::PositionGoal();
240  torso_goal->setLinkName("torso_lift_link");
241  torso_goal->setWeight(1);
242  torso_goal->setPosition(tf::Vector3( -0.05, 0, 1.0 ));
243  ik_options.goals.emplace_back(torso_goal);
244  ```
245 
246 For the actual turning motion, we calculate a set of required gripper
247 poses in a loop:
248  ```
249  for(int i = 0; ; i++) {
250  tf::Vector3 center(0.7, 0, 1);
251 
252  double t = i * 0.1;
253  double r = 0.1;
254  double a = sin(t) * 1;
255  double dx = fmin(0.0, cos(t) * -0.1);
256  double dy = cos(a) * r;
257  double dz = sin(a) * r;
258 
259  tf::Vector3 dl(dx, +dy, +dz);
260  tf::Vector3 dr(dx, -dy, -dz);
261  tf::Vector3 dg = tf::Vector3(0, cos(a), sin(a)) * (0.025 + fmin(0.025, fmax(0.0, cos(t) * 0.1)));
262 
263  ll_goal->setPosition(center + dl + dg);
264  lr_goal->setPosition(center + dl - dg);
265  rl_goal->setPosition(center + dr + dg);
266  rr_goal->setPosition(center + dr - dg);
267 
268  double ro = 0;
269  ll_goal->setOrientation(tf::Quaternion(tf::Vector3(1, 0, 0), a + ro));
270  lr_goal->setOrientation(tf::Quaternion(tf::Vector3(1, 0, 0), a + ro));
271  rl_goal->setOrientation(tf::Quaternion(tf::Vector3(1, 0, 0), a + ro));
272  rr_goal->setOrientation(tf::Quaternion(tf::Vector3(1, 0, 0), a + ro));
273 
274  lookat_goal->setAxis(tf::Vector3(1, 0, 0));
275  lookat_goal->setTarget(rr_goal->getPosition());
276 
277  // "advanced" bio-ik usage. The call parameters for the end-effector
278  // poses and end-effector link names are left empty; instead the
279  // requested goals and weights are passed via the ik_options object.
280  //
281  robot_state.setFromIK(
282  joint_model_group, // active PR2 joints
283  EigenSTL::vector_Affine3d(), // no explicit poses here
284  std::vector<std::string>(), // no end effector links here
285  0, 0.0, // take values from YAML file
286  moveit::core::GroupStateValidityCallbackFn(),
287  ik_options // four gripper goals and secondary goals
288  );
289 
290  ... // check solution validity and actually move the robot
291  }
292  ```
293 
294 When you execute the code, the PR2 will reach for the valve wheel
295 and turn it. Every once in a while it can't reach the valve with
296 its current arm configuration and will regrasp the wheel.
297

302 ## Local vs. Global Optimization, Redundancy Resolution, Cartesian Jogging
303 BioIK has been developed to efficiently find good solutions for non-convex inverse kinematics problems with multiple goals and local minima.
304 However, for some applications, this can lead to unintuitive results.
305 If there are multiple possible solutions to a given IK problem, and if the user has not explicitly specified which one to choose, a result may be selected randomly from the set of all valid solutions.
306 When incrementally tracking a cartesian path, this can result in unwanted jumps, shaking, etc.
307 To incrementally generate a smooth trajectory using BioIK, the desired behaviour should be specified explicitly, which can be done in two ways.
308 
309 #### Disabling Global Optimization
310 BioIK offers a number of different solvers, including global optimizers and local optimizers.
311 By default, BioIK uses a memetic global optimizer (`bio2_memetic`).
312 A different solver class can be selected by setting the `mode` parameter in the `kinematics.yaml` file of your MoveIt robot configuration.
313 
314 Example:
315 ```
316 all:
317  kinematics_solver: bio_ik/BioIKKinematicsPlugin
318  kinematics_solver_search_resolution: 0.005
319  kinematics_solver_timeout: 0.02
320  kinematics_solver_attempts: 1
321  mode: gd_c
322 ```
323 
324 Currently available local optimizers:
325 ```
326 gd, gd_2, gd_4, gd_8
327 gd_r, gd_r_2, gd_r_4, gd_r_8
328 gd_c, gd_c_2, gd_c_4, gd_c_8
329 jac, jac_2, jac_4, jac_8
330 ```
331 
332 Naming convention: `<solver type>_[<variant>_]<number of threads>`
333 
334 Notes:
335 - The `gd_*` solvers support arbitrary goal types.
336 - The `jac_*` solvers only support pose goals but might in theory be more stable in some cases.
337 - Relative performance depends on the application (it's probably best if you try it yourself for your particular robot and problem types).
338 - During our tests, the `gd_c` variants usually outperformed the other local solvers.
339 - For incremental tracking, a single-threaded variant without restarts is probably best suited (`gd_c`, `gd`, or `jac`).
340 - You can also create different MoveIt move groups with different solver types. If you now want to plan a cartesian trajectory from end effector pose A to end effector pose B, you can use a move group with a global optimizer to find a matching initial pose for end effector pose A and a different move group with a local optimizer for incrementally generating a smooth cartesian trajectory from A to B.
341 
342 #### Regularization
343 
344 You can force a global optimizer to return a local minimum through regularization.
345 
346 * For the specific case of incremental robot motions (aka *jogging*),
347  the simplest solution is to specify both a *PoseGoal* and the
348  special *RegularizationGoal*, which tries to keep the joint-space
349  IK solution as close as possible to the given robot seed configuration.
350  Typically you would use a high weight for the *PoseGoal* and a
351  smaller weight to the regularizer.
352 
353 * You can also add a *MinimalDisplacementGoal* instead of the
354  *RegularizationGoal*. Both goals try to keep the IK solution close
355  to the current/seed robot state, but differ slightly in the handling
356  of fast and slow robot joints (e.g. the PR2 has fast arm joints
357  and a rather slow torso lift joint).
358  You might want to play with both goals to see which one better
359  matches your needs.
360 
361 * Some industrial robot controllers on 7-DOF arms behave as if working
362  on a 6-DOF arm with one extra joint. Typically, the value of the extra
363  joint can be specified, and an IK solution is then searched for the
364  remaining six joints.
365  This behaviour can be achieved in bio-ik by combining a *PoseGoal*
366  for the end-effector with a *JointPositionGoal* for one (any one)
367  of the robot joints.
368 
369 * Another useful trick is trying to keep the robot joints centered,
370  as this will allow robot (joint) motions in both directions.
371  Just combine the *PoseGoal* with a *CenterJointsGoal*, and optionally
372  also a *RegularizationGaol*.
373 
374 * You can also combine regularization with a local `gd_*` solver.

378 ## How it works
379 
380 The bio-ik solver is based on a memetic algorithm that combines
381 gradient-based optimization with genetic and particle swarm optimization.
382 
383 Internally, vectors of all robot joint values are used to encode
384 different intermediate solutions (the *genotype* of the genetic algorithm).
385 During the optimization, joint values are always checked against the
386 active lower and upper joint limits, so that only valid robot configurations
387 are generated.
388 
389 To calculate the fitness of individuals, the cumulative error over all
390 given individual goals is calculated.
391 Any individual with zero error is an exact solution for the IK problem,
392 while individuals with small error correspond to approximate solutions.
393 
394 Individuals are sorted by their fitness, and gradient-based optimization
395 is tried on the best few configuration, resulting in fast convergence
396 and good performance for many problems.
397 If no solution is found from the gradient-based optimization,
398 new individuals are created by a set of mutation and recombination operators,
399 resulting in good search-space exploration.
400 
401 See [3] and [4] for more details. See [5] and [6] for an in-depth explanation of an
402 earlier evolutionary algorithm for animating video game characters.

404 ## Running the Self-Tests
405 
406 We have tested bio-ik on many different robot arms,
407 both using the tranditional single end-effector API
408 and the advanced multi end-effector API based on the KinematicsQueryOptions.
409 
410 One simple selftest consists of generating random valid robot configurations,
411 running forward kinematics to calculate the resulting end-effector pose,
412 and the querying the IK plugin to find a suitable robot joint configuration.
413 Success is then checked by running forrward kinematics again and checking
414 that the end-effector pose for the generated IK solution matches the target pose.
415 This approach can be run easily for thousands or millions of random poses,
416 samples the full workspace of the robot,
417 and allows to quickly generate success-rate and solution-time estimates
418 for the selected IK solver.
419 
420 Of course, running the tests requires installing the corresponding robot
421 models and adds a lot of dependencies.
422 Therefore, those tests are not included in the standard bio-ik package,
423 but are packaged separately.
424 
425 For convenience, we provide the `pr2_bioik_moveit` package,
426 which also includes a few bio-ik demos for the PR2 service robot.
427 These are kinematics only demos; but of course you can also try
428 running the demos on the real robot (if you have one) or the Gazebo
429 simulator (if you installed Gazebo).
